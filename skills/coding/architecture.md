# 架构设计技能 (Architecture Design)

## 描述
系统架构、模块划分、设计模式、技术选型

## 使用场景
- 新项目启动需要架构设计
- 系统重构需要重新规划
- 技术选型需要评估对比
- 性能瓶颈需要架构优化

## 架构设计原则

### SOLID 原则
| 原则 | 含义 | 应用 |
|------|------|------|
| **S**RP | 单一职责 | 一个类/模块只做一件事 |
| **O**CP | 开闭原则 | 对扩展开放，对修改关闭 |
| **L**SP | 里氏替换 | 子类可替换父类 |
| **I**SP | 接口隔离 | 接口要小而专 |
| **D**IP | 依赖倒置 | 依赖抽象而非具体 |

### 其他核心原则
- **KISS**: Keep It Simple, Stupid (保持简单)
- **DRY**: Don't Repeat Yourself (避免重复)
- **YAGNI**: You Aren't Gonna Need It (不要过度设计)
- **分离关注点**: 不同逻辑分开处理
- **最小知识原则**: 模块间减少耦合

## 架构模式

### 1. 分层架构 (Layered Architecture)
```
┌─────────────────┐
│   Presentation  │  ← UI/API 层
├─────────────────┤
│    Business     │  ← 业务逻辑层
├─────────────────┤
│     Service     │  ← 服务层
├─────────────────┤
│  Data Access    │  ← 数据访问层
├─────────────────┤
│    Database     │  ← 数据存储层
└─────────────────┘
```

**适用场景**: 传统企业应用、Web 应用
**优点**: 结构清晰、易于理解、职责分离
**缺点**: 层级过多可能影响性能

### 2. 微服务架构 (Microservices)
```
                    ┌──────────────┐
                    │   API Gateway│
                    └──────┬───────┘
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │   User      │ │   Order     │ │   Product   │
    │   Service   │ │   Service   │ │   Service   │
    └──────┬──────┘ └──────┬──────┘ └──────┬──────┘
           │               │               │
           ▼               ▼               ▼
    ┌─────────────┐ ┌─────────────┐ ┌─────────────┐
    │   User DB   │ │   Order DB  │ │  Product DB │
    └─────────────┘ └─────────────┘ └─────────────┘
```

**适用场景**: 大型系统、高并发、快速迭代
**优点**: 独立部署、技术异构、容错性好
**缺点**: 运维复杂、分布式事务、网络延迟

### 3. 事件驱动架构 (Event-Driven)
```
┌──────────┐     ┌─────────────┐     ┌──────────┐
│ Producer │ ──→ │   Message   │ ──→ │ Consumer │
│          │     │    Broker   │     │          │
└──────────┘     │  (Kafka/    │     └──────────┘
                 │   RabbitMQ) │
┌──────────┐     └─────────────┘     ┌──────────┐
│ Producer │ ──→                     │ Consumer │
└──────────┘                         └──────────┘
```

**适用场景**: 异步处理、解耦系统、实时流处理
**优点**: 高度解耦、可扩展、响应式
**缺点**: 最终一致性、调试困难、消息丢失风险

### 4. 六边形架构 (Hexagonal/Ports & Adapters)
```
            ┌──────────────────────┐
            │      Adapters        │
            │  (UI/API/DB/External)│
            └──────────┬───────────┘
                       │
            ┌──────────▼───────────┐
            │      Ports           │
            │   (Interfaces)       │
            └──────────┬───────────┘
                       │
            ┌──────────▼───────────┐
            │   Domain Logic       │
            │   (Core Business)    │
            └──────────────────────┘
```

**适用场景**: 领域驱动设计、测试驱动开发
**优点**: 核心业务独立、易测试、技术无关
**缺点**: 初期设计复杂、需要纪律

## 设计模式速查

### 创建型模式
| 模式 | 用途 | 示例 |
|------|------|------|
| Singleton | 保证单一实例 | 配置管理器、数据库连接池 |
| Factory | 创建对象不暴露逻辑 | 跨平台 UI 组件创建 |
| Builder | 复杂对象分步构建 | SQL 查询构建器 |
| Prototype | 通过复制创建对象 | 对象克隆、深拷贝 |

### 结构型模式
| 模式 | 用途 | 示例 |
|------|------|------|
| Adapter | 接口转换 | 第三方库适配 |
| Decorator | 动态添加功能 | Java IO 流、React HOC |
| Facade | 提供简化接口 | 子系统统一入口 |
| Proxy | 控制访问 | 懒加载、权限控制 |

### 行为型模式
| 模式 | 用途 | 示例 |
|------|------|------|
| Strategy | 可互换的算法 | 排序策略、支付策略 |
| Observer | 一对多依赖 | 事件系统、MVVM |
| Command | 请求封装为对象 | 命令队列、撤销操作 |
| Template | 定义算法骨架 | 框架钩子函数 |

## 技术选型框架

### 评估维度
```
1. 功能性
   - 是否满足需求？
   - 功能完整性如何？

2. 性能
   - 吞吐量、延迟
   - 资源消耗
   - 扩展性

3. 可靠性
   - 稳定性记录
   - 容错能力
   - 恢复机制

4. 可维护性
   - 代码质量
   - 文档完整性
   - 社区活跃度

5. 成本
   - 许可费用
   - 学习成本
   - 运维成本

6. 风险
   - 供应商锁定
   - 技术债务
   - 安全风险
```

### 决策矩阵示例
| 选项 | 功能 | 性能 | 维护 | 成本 | 风险 | 总分 |
|------|------|------|------|------|------|------|
| A | 9 | 7 | 8 | 6 | 7 | 37 |
| B | 7 | 9 | 7 | 8 | 8 | 39 ← |
| C | 8 | 6 | 9 | 7 | 6 | 36 |
