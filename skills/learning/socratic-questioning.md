# 苏格拉底式提问技能 (Socratic Questioning)

## 描述
通过问答引导深度理解，激发思考而非直接给答案

## 使用场景
- 复习已学知识
- 探索新概念
- 深度讨论
- 检验理解程度
- 引导自主发现

## 提问类型

### 1. 澄清问题 (Clarification)
**目的**: 确保理解概念的本质

**示例**:
- "你说的 X 是什么意思？"
- "能否用简单的话解释 X？"
- "X 的核心是什么？"
- "这和 Y 有什么区别？"

**应用**:
```
用户：我想学习机器学习

助手：好的。在你看来，"机器学习"的核心是什么？
它和传统编程有什么本质区别？
```

### 2. 检验假设 (Assumptions)
**目的**: 揭示隐含的前提和假设

**示例**:
- "这个结论基于什么假设？"
- "为什么假设 X 成立？"
- "如果 X 不成立会怎样？"
- "有什么证据支持这个假设？"

**应用**:
```
用户：深度学习一定比传统机器学习好

助手：这个观点基于什么假设？
在什么情况下深度学习可能不如传统方法？
你能想到反例吗？
```

### 3. 检验证据 (Evidence)
**目的**: 评估观点和结论的可靠性

**示例**:
- "有什么证据支持这个观点？"
- "这个证据可靠吗？为什么？"
- "有相反的证据吗？"
- "证据是否充分？"

**应用**:
```
用户：这个模型效果很好

助手：你如何判断"效果很好"？
用了什么评估指标？
测试集是什么？
有没有可能过拟合？
```

### 4. 探索视角 (Perspectives)
**目的**: 从不同角度思考问题

**示例**:
- "从 X 的角度会怎么看？"
- "相反的观点是什么？"
- "如果站在 Y 的立场呢？"
- "还有其他解释吗？"

**应用**:
```
用户：应该用微服务架构

助手：支持微服务的理由是什么？
反对微服务的观点有哪些？
如果团队只有 3 个人，还适合微服务吗？
从运维角度看呢？
```

### 5. 检验后果 (Consequences)
**目的**: 理解决策和观点的影响

**示例**:
- "这样做的后果是什么？"
- "短期和长期影响分别是什么？"
- "有什么风险？"
- "如果错了，代价是什么？"

**应用**:
```
用户：我们要重构整个系统

助手：重构的收益是什么？
成本是什么？
如果重构失败，影响有多大？
有没有渐进式方案？
```

### 6. 元问题 (Meta-Questions)
**目的**: 对思考过程本身的反思

**示例**:
- "我们刚才的推理有漏洞吗？"
- "这个问题本身有问题吗？"
- "我们忽略了什么？"
- "如何验证我们的结论？"

**应用**:
```
用户：所以 A 方案比 B 方案好

助手：我们得出这个结论的推理过程严谨吗？
有没有考虑所有因素？
我们的评估标准是否合理？
如何验证这个结论？
```

## 提问流程

### 1. 探索阶段 (Exploration)
**目标**: 了解用户的当前理解

```
助手：关于 X，你现在知道什么？
用户：[回答]

助手：哪里还不清楚？
用户：[回答]
```

### 2. 挑战阶段 (Challenge)
**目标**: 检验理解的深度

```
助手：为什么这样认为？
用户：[回答]

助手：如果 [反例场景] 呢？
用户：[回答]
```

### 3. 重构阶段 (Reformulation)
**目标**: 帮助形成更清晰的理解

```
助手：所以你的意思是...[重述]？
用户：对/不对，其实是...

助手：那这和之前说的 Y 有什么关系？
用户：[建立联系]
```

### 4. 总结阶段 (Summary)
**目标**: 巩固学习成果

```
助手：回顾一下，我们今天发现了什么？
用户：[总结]

助手：还有什么疑问吗？
用户：[提问]
```

## 提问技巧

### 1. 由浅入深
```
Level 1: 是什么？(定义)
Level 2: 为什么？(原理)
Level 3: 怎么用？(应用)
Level 4: 如果...？(变体)
Level 5: 本质？ (第一性原理)
```

### 2. 等待时间
- 提问后给用户思考时间
- 不要急于补充或纠正
- 允许沉默

### 3. 追问技巧
```
用户：机器学习就是让电脑变聪明

助手：[追问] "变聪明"具体指什么？
能举个例子吗？
这个例子中，电脑"学"的是什么？
```

### 4. 引导而非告知
```
❌ 告知：机器学习的定义是...
✅ 引导：你觉得机器学习和普通程序有什么区别？
         程序是怎么"学会"新技能的？
```

## 问题模板库

### 概念理解
- "X 的本质是什么？"
- "X 和 Y 的核心区别？"
- "没有 X 会怎样？"
- "X 解决了什么问题？"

### 原理探索
- "为什么 X 有效？"
- "X 的工作原理？"
- "X 的局限性？"
- "X 的前提条件？"

### 应用实践
- "什么时候用 X？"
- "怎么用 X 解决 Y？"
- "X 的实际案例？"
- "用 X 时要注意什么？"

### 深度思考
- "X 的第一性原理？"
- "X 的底层假设？"
- "如果重新设计 X？"
- "X 的未来发展？"

## 示例对话

### 示例 1: 复习机器学习

**助手**: 我们来复习一下机器学习。

**第一个问题**: 机器学习的核心是什么？
用一句话概括。

**用户**: 就是从数据中学习规律

**助手**: 好。那这个"规律"具体指什么？
能举个例子说明吗？

**用户**: 比如从邮件数据中学习什么是垃圾邮件

**助手**: 很好。那在这个例子中：
1. "数据"是什么？
2. "规律"是什么形式？
3. 怎么判断学到的"规律"好不好？

**用户**: 数据是邮件内容和标签，规律是分类规则，用准确率判断

**助手**: 准确。那如果准确率高但实际效果不好，可能是什么原因？

**用户**: 可能是数据不平衡？

**助手**: 对。这引出了什么重要概念？

**用户**: 评估指标不能只看准确率，还要看召回率、F1 值

**助手**: 很好！你把这个概念和实际问题联系起来了。

### 示例 2: 讨论技术选型

**用户**: 我觉得应该用微服务

**助手**: 好的。支持微服务的主要理由是什么？

**用户**: 可扩展性好，团队可以独立开发

**助手**: 这是微服务的优势。那代价是什么？
独立开发带来的复杂度增加，如何权衡？

**用户**: 确实，运维会变复杂

**助手**: 具体来说，哪些运维工作会变复杂？
如果团队没有专职运维，会影响吗？

**用户**: 部署、监控、日志都会变复杂...

**助手**: 那在什么情况下，微服务的收益大于成本？
什么情况下应该用单体架构？

**用户**: 团队大、业务复杂时用微服务，否则用单体

**助手**: 很好。那我们现在的团队规模和业务复杂度如何？
这个判断标准适用吗？
